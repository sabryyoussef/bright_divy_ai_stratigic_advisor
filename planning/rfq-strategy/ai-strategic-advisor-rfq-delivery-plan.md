# خطة تسليم عملية منخفضة المخاطر — مشروع «تطوير وتكامل المستشار الاستراتيجي بالذكاء الاصطناعي»

**الإصدار:** 1.0  
**التاريخ:** 2026-04-26  
**الحالة:** مسودة تنفيذية

---

## 1. الفهم التجاري

### ما هو طلب العروض (RFQ) ببساطة؟

العميل يطلب **نظام محادثة ذكي مغلق المصدر ومستضاف ذاتياً** لمدير تنفيذي في أكاديمية قيادة محلية. المدير يطرح أسئلة بالإنجليزية أو العربية ويحصل على **رؤى استراتيجية** مبنية على بيانات من ثلاثة مصادر: Tally للمالية، Adler لـ HRIS، وجداول Google لملاحظات العملاء. البيانات تُجمع **دُفعات يومية أو أسبوعية** إلى مستودع بيانات سحابي مؤقت؛ في الطيار يُقبل **تصدير CSV يدوي**.

### ماذا يريد العميل فعلاً؟

| الطلب | المعنى العملي |
|--------|----------------|
| تحليل أداء مالي | مؤشرات وصيغ محددة من Tally/التصديرات |
| مخاطر تشغيلية من تعليقات الطلاب | تحليل نص عربي/إنجليزي (أعلى تعقيداً في الطيار) |
| مقارنة أداء (Benchmarking) | تعريف معيار ومقارنة فعلية بالبيانات |
| واجهة محادثة | تطبيق محادثة مع أزرار إجراءات (المرحلة 1: تأكيد واجهة فقط) |
| ذكاء اصطناعي ذاتي الاستضافة | لا OpenAI ولا Google كخدمات عامة للنماذج |
| سحابة خاصة / VPC مخصص | عزل شبكة وبيانات |
| RBAC + تدقيق | أدوار، سجلات، مساءلة |
| امتثال PDPPL قطر وإرشادات NCSA للذكاء الاصطناعي | تصميم + تشغيل + سياسات، وليس «ميزة برمجية» واحدة |

### المرحلة 1 مقابل المرحلة 2

| | **المرحلة 1 (Must Have للطيار 60 يوماً)** | **المرحلة 2 (Nice to Have / لاحقاً)** |
|---|-------------------------------------------|----------------------------------------|
| التركيز | **التحليل المالي الاستراتيجي** حسب مواصفات وصيغ واضحة | تحليل مخاطر تشغيلية من تعليقات عربية/إنجليزية + معايير F1 وما شابه |
| البيانات | CSV يومي/أسبوعي من Tally (مصدر واحد كافٍ للبداية) | تكامل Adler + Google Sheets + جدولة n8n إن لزم |
| الحسابات | Python/FastAPI + PostgreSQL (حتمي، قابل للتدقيق) | نماذج NLP أثقل، معايرة دقة عربية |
| الواجهة | Dify: محادثة + أزرار Phase 1 (تأكيد UI) | توسيع الإجراءات والتكامل الحقيقي مع الأنظمة |

### ما لا نعد به قبل توقيع NDA

- **صيغ نهائية** أو تعريفات I/O كاملة للاختبارات.
- **سعر ثابت نهائي** أو التزام بمدة/نطاق دون FSD.
- **ضمان دقة عربية محددة (مثل 85% F1)** قبل وجود مجموعة اختبار معتمدة من العميل.
- **تفاصيل تكامل Tally/Adler** (حقول، APIs، جداول) دون وصول للأنظمة أو الوثائق الرسمية.
- **تصميم امتثال نهائي** كـ «شهادة»؛ نقدم **إطار امتثال وتصميم** مع افتراضات واضحة.

## 2. استراتيجية التسليم

### لماذا نبدأ بـ Expression of Interest وليس العرض الكامل؟

- العرض الكامل يتطلب **FSD** (صيغ، سيناريوهات، بيانات عينة، مجموعة اختبار عربية) — وهذا **بعد NDA**.
- EOI يُظهر **جدية وجاهزية** دون التزامات تقنية/مالية مفرطة.
- يقلل مخاطرة **التسعير الخاطئ** بسبب غموض التكامل والصيغ.

### لماذا التسعير النهائي يعتمد على FSD؟

| بدون FSD | مع FSD |
|----------|--------|
| تخمين الصيغ والمؤشرات | تعريف حتمي للمدخلات/المخرجات |
| عدم معرفة حجم البيانات والجودة | سيناريوهات اختبار قابلة للتوقيع عليها |
| خطر نطاق غير محدود | نطاق Phase 1 قابل للإغلاق |

### التموضع: مزود تكامل بيانات + ذكاء اصطناعي حوكَم

- نؤطر الحل كـ: **مستودع بيانات + محرك تحليلات موثوق + طبقة محادثة حوكَمة**.
- نبرز: **تدقيق الحسابات خارج LLM**، **RBAC**، **سجلات تدقيق**، **نشر خاص**.

### ذكر Dify دون أن يبدو «تثبيت Dify»

| Avoid | Recommend |
|-------|-----------|
| «الحل هو Dify» | «طبقة تطبيق محادثة وسير عمل **مستضافة ذاتياً** (مثل Dify) تستدعي **خدمات تحليلات موثوقة** عبر HTTP» |
| تخزين أعمال في Dify | «البيانات التجارية في **PostgreSQL منفصل**؛ Dify للتوجيه والصياغة فقط» |

## 3. خطة المعمارية

### المعمارية الموصى بها (نص مخطط)

**المدير التنفيذي → تطبيق محادثة Dify (Docker) → Workflow/Agent في Dify → استدعاء أداة HTTP → خدمة FastAPI (تحليلات حتمية) → PostgreSQL (مستودع أعمال) → JSON موثّق → Dify + LLM محلي لصياغة الإجابة فقط.**

### المكونات ومسؤولياتها

| المكون | المسؤولية | Must Have / Nice to Have |
|--------|-----------|---------------------------|
| **Dify** | تجربة محادثة، سير عمل، استدعاء الأدوات، (أزرار Phase 1 حسب إمكانيات المنصة) | Must Have |
| **FastAPI** | كل الحسابات، القواعد، التجميع، التحقق، تنسيق JSON للاستجابة | Must Have |
| **PostgreSQL (مستودع أعمال منفصل)** | جداول حقيقة (facts)، أبعاد بسيطة، جداول تدقيق مرجعية للقراءة من التطبيق | Must Have |
| **LLM محلي (Ollama أو vLLM)** | فهم السؤال، اختيار الأداة/المعاملات تقريباً، صياغة النص بالعربية/الإنجليزية **من أرقام جاهزة** | Must Have (Ollama أبسط للبداية) |
| **n8n** | جدولة استيراد CSV لاحقاً | Nice to Have |
| **سحابة خاصة / VPC** | عزل، جدار نار، VPN، أسرار، نسخ احتياطي | Must Have (للإنتاج/الطيار المعتمد) |

### ما لا يجوز أن يتولاه Dify أو الـ LLM

| Avoid | Must |
|-------|------|
| حساب الإيرادات، الهامش، «النزيف المالي»، أي KPI رقمي | FastAPI + SQL/Python |
| تخزين بيانات أعمال حساسة في قاعدة Dify الداخلية | PostgreSQL منفصل؛ Dify يحتفظ بالحد الأدنى من السياق حسب السياسة |
| «استنتاج» أرقام من نص غير مُحقق | الأرقام من API موقعة/مخططة JSON من الخادم |

## 4. خطة النموذج المحلي (Prototype)

### خطوات إعداد محلي (Docker حيث أمكن)

1. **PostgreSQL** (حاوية) + إنشاء قاعدة `warehouse` وجداول بسيطة (مثلاً `fact_transactions`, `dim_date`).
2. **تحميل CSV مالي عينة** (استيراد `COPY` أو سكربت صغير).
3. **FastAPI**: endpoint واحد مثلاً `POST /analytics/financial-bleed` يقرأ من PostgreSQL ويعيد JSON ثابت الشكل.
4. **Ollama** (محلي) بنموذج متوسط الحجم **CPU** (بدون GPU في البداية).
5. **Dify** (Docker): تطبيق محادثة + Workflow يستدعي FastAPI كـ **HTTP Tool**، مع تعليمات نظام: «لا تخمن أرقاماً؛ استخدم JSON فقط».
6. اختبار سؤال إنجليزي وعربي لنفس المنطق.

### حالة استخدام دنيا: «Show financial bleed» / «أظهر النزيف المالي**

**تعريف بسيط للعينة:** نزيف = صافي تدفق سلبي متكرر أو تكاليف تتجاوز إيرادات فترة محددة (يُستبدل بتعريف FSD).

**عينة مدخلات (CSV مبسط):**

| date | account_type | amount_qar | category |
|------|--------------|------------|----------|
| 2026-01-15 | revenue | 50000 | lessons |
| 2026-01-20 | expense | 62000 | payroll |

**استجابة API متوقعة (JSON):**

```json
{
  "period": "2026-01",
  "net_qar": -12000,
  "bleed_detected": true,
  "summary_key": "NEGATIVE_NET",
  "drivers": [{"category": "payroll", "impact_qar": -62000}]
}
```

**مخرج محادثة متوقع (إنجليزي):**  
Net for January 2026 is **-12,000 QAR**. Bleed is driven mainly by **payroll** relative to **lesson revenue**.

**مخرج محادثة متوقع (عربي):**  
صافي شهر يناير 2026 هو **-12,000 ر.ق**. يظهر نزيف مالي مدفوعاً أساساً بـ **رواتب** مقارنة بإيرادات **الدروس**.  
(الأرقام من JSON فقط؛ الصياغة من LLM.)

## 7. الامتثال والأمن

| المتطلب | التنفيذ |
|---------|---------|
| لا LLM عام | نشر Ollama/vLLM داخل VPC؛ لا مفاتيح لـ OpenAI/Google لمعالجة بيانات الإنتاج |
| نشر خاص | VPC، عناوين خاصة، وصول VPN أو Zero Trust حسب سياسة العميل |
| RBAC | أدوار CEO/مسؤول تقني/قراءة فقط؛ فصل وصول Dify عن وصول قاعدة البيانات |
| تدقيق | تسجيل: من طلب ماذا، أي endpoint، الطابع الزمني، معرف الجلسة (بدون تخزين أسرار في السجلات) |
| تشفير ونسخ احتياطي | TLS داخلياً حيث ينطبق؛ تشفير أقراص السحابة؛ نسخ احتياطي PostgreSQL مجدول |
| PDPPL / NCSA | **إطار**: تصنيف بيانات، تقليل الاحتفاظ، DPA، DPIA عند الحاجة، سياسات بشرية + تقنية — **ليس ادعاء شهادة برمجية وحدها** |

## 9. استراتيجية التسعير (مؤشرات — ليس سعراً نهائياً قبل NDA)

**نطاق إرشادي Phase 1 (ر.ق):** **180,000 – 450,000 ر.ق** تقريباً للطيار 60 يوماً — **قابل للتعديل كلياً بعد FSD**.  
**Phase 2** تُقتبس **منفصلة**.

| بند | Phase 1 | ملاحظة |
|-----|---------|--------|
| إعداد مستودع + أمن أساسي | ثابت نسبياً | Must |
| تكامل لكل نظام | لكل مصدر / اختياري حتى يتضح FSD | اختياري/متدرج |
| «تطوير نموذج AI» | Phase 1: توجيه + أدوات + ضوابط؛ NLP ثقيل في Phase 2 | فصل واضح |
| واجهة محادثة/لوحة | Dify + ضبط | Must |
| اختبار/نشر | ضمن النطاق | Must |
| تدريب/توثيق | بند ثابت صغير/متوسط | Must |
| صيانة | شهرياً اختياري | Nice / اختياري |

**ثابت مقابل اختياري:** ثابت ضمن حدود FSD: وحدة مالية Phase 1، نشر طيار واحد، UAT محدود. اختياري: تكامل Adler، Google Sheets، n8n، توسيع NLP.

**افتراضات حماية:** CSV متاح وجدولي؛ تأخير موافقات العميل خارج النطاق؛ تغيير الصيغ = تغيير نطاق؛ لا ضمان F1 قبل مجموعة اختبار معتمدة.

## 10. مسودات جاهزة للعميل

### أ) بريد Expression of Interest (EOI)

**الموضوع:** Expression of Interest — AI Strategic Advisor (Self-Hosted)

السادة في NexSolve،  
نود التعبير عن اهتمامنا بتقديم عرض لمشروع «تطوير وتكامل المستشار الاستراتيجي بالذكاء الاصطناعي» لصالح أكاديمية القيادة.  
نتخصص في تكامل البيانات، مستودعات البيانات، ونشر حلول ذكاء اصطناعي **مستضافة ذاتياً** مع **فصل صارم** بين طبقة المحادثة و**محرك تحليلات حتمي** يضمن قابلية التدقيق وعدم اعتماد النماذج اللغوية على الحسابات الرقمية.  
نرفق ملفاً تعريفياً مختصراً ودراستي حالة ذات صلة. نرحب بتوقيع الاتفاقية المتبادلة لعدم الإفصاح (NDA) تمهيداً لاستلام وثيقة المواصفات الوظيفية (FSD) التي تحدد الصيغ وسيناريوهات الاختبار والبيانات العينة؛ وبعدها نقدم العرض الفني الكامل (حتى 10 صفحات) والتسعير التفصيلي.

### ب) رسالة قصيرة (واتساب / لينكدإن)

اهتمام رسمي بمشروع المستشار الاستراتيجي بالذكاء الاصطناعي (نشر ذاتي، بدون LLM عام). نرسل ملفاً تعريفياً ودراستي حالة؛ جاهزون لخطوة NDA ثم FSD للعرض التفصيلي.

### ج) فقرة Company Profile (قصيرة)

نحن [الشركة] نقدم حلول تكامل بيانات وتحليلات موثوقة مع طبقات محادثة **مستضافة ذاتياً** للقيادة التنفيذية. نركز على **حوكمة البيانات**، **RBAC**، **سجلات التدقيق**، و**فصل الحسابات عن النماذج اللغوية** لضمان إجابات قابلة للمراجعة في بيئات تنظيمية صارمة.

### د) دراستا حالة (إطار — عوّض بالحقائق الحقيقية)

1. تكامل ERP/مالية + مستودع + تقارير تنفيذية: تدفقات دُفعية، جودة بيانات، لوحة قرارات.  
2. منصة داخلية لاستعلامات معتمدة: API تحليلات حتمية + واجهة محادثة للإدارة العليا مع تدقيق.

### هـ) أسئلة توضيح بعد NDA

- تعريفات KPI والصيغ المعتمدة في FSD؟  
- تنسيق تصدير Tally (CSV) والحقول الإلزامية؟  
- لغة واجهة Adler وطريقة التصدير المتاحة؟  
- سياسة الاحتفاظ بالبيانات والتصنيف (شخصي/حساس)؟  
- VPC/مزود السحابة المفضل؟  
- أدوار المستخدمين المستهدفة لـ RBAC؟  
- متطلبات NCSA/PDPPL من جهة العميل (DPIA، DPA)؟

### و) ملخص تقني (Technical approach summary)

نعتمد **مستودع PostgreSQL منفصلاً** كمصدر حقيقة للبيانات التجارية، و**خدمة FastAPI** تنفذ جميع الحسابات والقواعد وتعيد **JSON موثّق**. طبقة **محادثة وسير عمل مستضافة ذاتياً** (مثل **Dify** على Docker) تتولى تجربة المستخدم واستدعاء الأدوات عبر HTTP، بينما **نموذج لغوي محلي** (مثل Ollama) يقتصر على فهم السؤال وصياغة الإجابة من الأرقام المعتمدة فقط، دون أي خدمات LLM عامة. يُفصل تخزين بيانات الأعمال عن قواعد بيانات تطبيق المحادثة لسهولة التدقيق والامتثال.

## الخاتمة التنفيذية

### خطة العمل للـ 24 ساعة القادمة

1. تجميع **Company profile** + **دراستا حالة** (حقيقيتان).  
2. رسم **مخطط نصي** للمعمارية (قابل للصق في العرض).  
3. كتابة **قائمة افتراضات** و**أسئلة ما بعد NDA**.  
4. إرسال **EOI** مع المرفقات.  
5. محلياً: تشغيل **PostgreSQL + FastAPI + endpoint واحد** ثم **Dify + استدعاء HTTP** (حتى بدون LLM في اليوم الأول إن لزم التسلسل).

### ماذا نرسل للعميل الآن

بريد **EOI** + **Company profile** + **دراستا حالة** + سطر واحد عن **النهج التقني** (بدون تسعير نهائي وبدون التزام بصيغ قبل FSD).

### ماذا نبني محلياً أولاً

**PostgreSQL + CSV عينة + FastAPI (`financial-bleed`) + Dify workflow يستدعي API + Ollama لصياغة ثنائية اللغة من JSON فقط.**
